Allowed memory size of 8388608 bytes exhausted

环     境：discuz 6.0

症     状：无法上传较大的图片附件，较大的其他附件以及较小的图片附件都没问题；中间偶尔出现一次

Fatal error: Allowed memory size of 8388608 bytes exhausted (tried to allocate 5760 bytes) in   /www/users/bhclub.net/include/image.class.php on line 179

问题确定：网上搜索，基本确定是分配给图片上传的内存不够用了，于是只能上传较小的附件

解决办法：登陆ftp，修改文件image.class.php，增加语句：ini_set(‘memory_limit’, ‘-1’);

参考文章：hi.baidu.com/us_hosting/blog/item/5e562d6d507e91f842169424.html

Windows（XP Vista）引导linux

用linux启动光盘的rescue功能在主引导分区重新安装GRUB。

grub-install /dev/hda

然后重新启动进入linux，在你的Linux主分区安装GRUB，比如说你Linux安装在/dev/hda5，那就是

grub-install /dev/hda5

将Linux的启动块复制为文件：

dd if=/dev/hda5 of=linux.bin bs=512 count=1

然后将获得的linux.bin文件复制到Windows文件系统的C:\下

下面分两种情况XP和Vista来具体操作：

XP

在C:\boot.ini最后添加下面一行：

C:\linux.bin="Linux"

最后用Windows系统盘fdisk /mbr恢复MBR，此时就能够用Windows启动Linux了。

Vista

bcdedit /create /d "Linux" /application bootsector

命令结果应该会返回一个GUID，后面的命令用到这个GUID

运行以下命令设置Linux引导扇区备份所在的分区：

bcdedit /set {GUID} device partition=C:

然后指定引导扇区备份的路径：

bcdedit /set {GUID} path linux.bs

最后设置Linux菜单项在启动菜单的位置：

bcdedit /displayorder {GUID} /addlast

永不放弃

原作者: A Christian Science perspective 译者: 译家之言 (Blog) 发表时间：14小时40分钟前 原文链接

[

](http://www.csmonitor.com/2007/1120/p18s01-hfcs.html)

在这惊险的一幕上演之前，一切看上去都很平静。一群非洲水牛，大约100只，安详地顺着河岸漫步。它 们没有为了安全而紧密地挤在一起；相反，它们四散开来，蜿蜒有数百米长。领头的公牛独自挑头，走在最前面。但是，它没有注意到，前方正潜伏着一群饥饿的狮 子。它仍然毫无知觉地带领群牛走向狮子的包围圈。狮子伏下身来，密切地注视着。突然，它们出击了！水牛群被突如其来的攻击吓得四散奔逃，仓皇逃窜。 刹那间，狮群收紧了包围，把目标锁定在一头掉队的幼年水牛上。不过几秒钟，狮子就将它扑倒在地，然后一同滚进河里。 狮子们试图将小水牛赶到河岸上。让人难以置信地，小牛居然进行了顽强的反击。就在此刻，一头在水面下 埋伏了许久的鳄鱼，也加入了这场战斗。当狮子试图将小牛拽向岸上的时候，鳄鱼紧紧地咬住了小牛的后腿。双方几经争夺，最后鳄鱼举手投降，眼睁睁地看着狮子 把到手的猎物夺了回去。 出人意料地，这时在岸上的水牛群重新集结起来，向狮子发动攻势。凭借有力的犄角，领头的公牛冲向一头狮子并将其挑到半空当中。狮群于是溃散开来，各自逃命。逃出狮口的小牛拖着跛脚，重新回到家族的怀抱。 这惊险的一幕，被一名业余摄影者用摄像机捕捉下来。现在该视频可以在YouTube上看到（http://www.youtube.com/watch?v=Vyqkzfznx6c）。 小水牛的奋争印证了一句话：永远不要放弃。哪怕是在最危急的时刻，小牛和它的家族也没有丝毫退让。 小牛的奋争和群牛的协助，让我联想到正在世界各地进行的数不清的战斗。许多的个体正在进行看似无望的 斗争，他们的目标，或是希望渺茫的顽疾、或是泛滥成灾的毒品、或是一贫如洗的匮乏、或是独裁暴虐的专制。而国家和地区之间的共同努力，则是针对饥荒、旱 灾、流行病、还有腐败统治。 “永远不要放弃”的普适性涵盖整个地球。同样的，它也适用于我们每一个人。 许多年以前，我认识的一位女士用平静无声的方式教会我，不论治愈的希望多么遥不可及，也不要放弃。我 碰到她的时候，她差不多95岁。近20年来，她都在一只眼睛的痛苦生活中煎熬。她试图依靠精神的方法来治愈她的疾病。虽然我并未开口，但是我猜想她为了治 病已经祈祷了无数次。 对于她的奢望，我敬佩有加，也甚至偷偷地怀疑，与她同样年龄的人，（面对如此情况）可能会想，“一只 眼睛、两只眼睛到底有多大区别呢？反正都过得去”。但是，她的理论不是这样的。她并未自暴自弃。她并未接受这样的说法，“除了上帝、圣灵和真爱之外的一切 事物，都处于完全控制之下”。她依然满怀信心地追求自然治愈和精神成长。经过若干时日，她的疾病居然因为念诵主祷文而痊愈了。 在她去世的时候，她已经安然度过了百岁生日。（关于她的疾病自然治愈的故事，请参阅1990年7月2日的《基督教科学看守人（Christian Science Sentinel）》，第17页） 来自本报创始人Mary Baker Eddy留下的一段话，尤其令人释怀：“Remember, thou canst be brought into no condition, be it ever so severe, where Love has not been before thee and where its tender lesson is not awaiting thee. Therefore despair not nor murmur, for that which seeketh to save, to heal, and to deliver, will guide thee, if thou seekest this guidance”（《The First Church of Christ, Scientist, and Miscellany》第149至150页）。 任何瞥见生命与真爱的时刻，都给我们以力量去面对治愈与自新，而不是放弃。每一次这样的瞬间都孕育了对个体与人性的希望。在未来的日子里，我希望自己依然记得这段视频。对于它所传递的信息，我希望能保留更长的时间。

Yahoo!的项目工作流程 zz

找到一份重要参考资料，采访时间是2006年2月。

王小丹：主要网络产品设计工作流程是这样的：

网络产品设计：产品制作人写产品计划书，确定新产品或新功能的市场意义和经济效益，提交部门审批，同意后，确认需要设计的部分，和用户体验研究员(user researcher)，信息建构师(information architect)，视觉设计师(visual designer)、user interface designer，互动设计师(interaction designer)，web developer，工程师(engineer)一起讨论需要的支持，然后订出时间计划分工合作。一般是先由用户体验研究员作调查、分析后由信息建构师设计产品架构，然后由互动设计师作出互动流程，之后交给视觉设计师(visual designer)和user interface designer作出视觉设计。然后web developer把设计通过编写程序(html, dhtml, JavaScript)等等再现出来，最后交给工程师。做完后用户体验研究员需要做用户测试，QA(Quality assurance) 需要测验这一产品的每一步骤，确认产品的使用质量，如果有问题需要让工程师或相关人员解决。雅虎欧洲设计师王小丹访谈

总结起来就是：

  1. 产品制作人，写产品计划书。
  2. 用户体验研究员，作调查分析。
  3. 信息建构师，设计产品结构。
  4. 互动设计师，作出互动流程。
  5. 视觉设计师和用户界面设计师，作出页面视觉设计。
  6. 前台工程师，前台开发。
  7. 后台工程师，后台开发。
  8. 用户体验研究员，做用户测试确保质量。

应该算是一个比较完善成熟的流程了，信息建构(IA)和互动设计(ID)明显处于很重要的位置。

国内项目普遍流程就是“策划＋设计＋制作＋开发”，普遍忽略的是第2, 3, 4，或者说是其他某个流程附带完成，第6前台开发也是只存在于大型正规公司，或者说有普遍进一步完善的趋势。

至于UE, IA, ID如何进行，我觉得没有绝对固定的模式，根据项目实际情况，关键在于团队的默契。不是说国内从业者不知道如何进行，而是根本没有这个习惯，把项目环节细分一下，就很清楚各自职能了。这些流程的核心目的就是逐步保证产品质量，减少“前台开发”和“后台开发”的重复劳动。

就Yahoo这个流程来看，应该是针对大型的应用型产品项目。看看人家在“确认设计需求”时，就需要几乎所有项目环节参与者进行综合讨论，让人狂汗不已；定开发计划都是大家讨论进行，而不是单方面强制决定，多么人性化啊。
